# Чеклист прогонки Phase 5.5

Ручной чеклист для проверки нового intake-режима GNB-бота в Telegram/stage перед дальнейшим движением.

## 0. Подготовка

- [x] Бот отвечает в Telegram
- [x] Stage-бот запущен
- [x] В памяти есть предыдущий ГНБ для тестового объекта
- [x] XLSX-шаблоны лежат на месте
- [x] Подготовлены тестовые файлы:
- [x] ИС в PDF
- [x] Паспорт трубы в PDF
- [x] При необходимости фото документа
- [x] При необходимости Excel

## 1. Старт нового ГНБ с базой

Шаги:
- [ ] Отправить `/new_gnb`
- [ ] Ввести компанию
- [ ] Ввести объект
- [ ] Ввести новый номер ГНБ

Проверка:
- [ ] Бот нашёл предыдущий ГНБ
- [ ] Бот спросил, использовать ли его как основу
- [ ] Ответ `да` перевёл бота в режим сбора данных
- [ ] Бот попросил прислать новые материалы по текущему ГНБ

## 2. Приём документов

Отправить:
- [ ] ИС PDF
- [ ] Паспорт трубы PDF
- [ ] Текстовые правки / даты / подписантов
- [ ] При необходимости фото
- [ ] При необходимости Excel

Проверка:
- [ ] Каждый документ ушёл именно в intake-режим
- [ ] Бот отвечает как intake, а не как обычный чат
- [ ] В ответе виден `doc_class`
- [ ] В ответе видно, какие поля обновились
- [ ] Черновик не теряется

## 3. Review

Шаги:
- [ ] Отправить `/review_gnb`

Проверка:
- [ ] Показан Паспорт ГНБ
- [ ] Показан блок унаследованных данных
- [ ] Показан блок изменённых данных
- [ ] Показан блок "требует проверки"
- [ ] Показан блок "чего не хватает"
- [ ] Если есть споры, показан блок конфликтов
- [ ] Если нет ИС, бот явно требует схему или геометрию текстом

## 4. Проверка блокировки при неполных данных

Шаги:
- [ ] Оставить черновик заведомо неполным
- [ ] Попробовать подтвердить

Проверка:
- [ ] Подтверждение блокируется, если нет обязательных полей
- [ ] Бот не генерирует файлы
- [ ] Бот понятно пишет, чего не хватает
- [ ] Черновик остаётся доступным

## 5. Проверка конфликтов

Шаги:
- [ ] Подать конфликтующие данные: адрес / шифр / подписант / паспорт
- [ ] Снова вызвать `/review_gnb`

Проверка:
- [ ] Бот не выбирает значение автоматически
- [ ] Конфликт виден в review
- [ ] Показаны оба значения и их источники
- [ ] Подтверждение блокируется, пока конфликт не решён

## 6. Успешное подтверждение

Шаги:
- [ ] Довести черновик до полного состояния
- [ ] Подтвердить

Проверка:
- [ ] Черновик финализируется
- [ ] Transition сохраняется
- [ ] Генерируются 2 файла
- [ ] В Telegram приходит XLSX с актами
- [ ] В Telegram приходит XLSX с АОСР
- [ ] Финальное сообщение корректное

## 7. Проверка сгенерированных файлов

Проверка:
- [ ] В обоих файлах одинаковый номер ГНБ
- [ ] В обоих файлах одинаковый адрес
- [ ] Длины корректные
- [ ] Подписанты корректные
- [ ] Данные по трубе корректные
- [ ] Нет явных остатков шаблона
- [ ] АОСР не развален по формулам/ссылкам между листами

## 8. Проверка resume

Шаги:
- [ ] Прервать intake на середине
- [ ] Снова отправить `/new_gnb`

Проверка:
- [ ] Бот предлагает продолжить / отменить / начать заново
- [ ] Resume возвращает в тот же черновик
- [ ] `/review_gnb` показывает уже собранные данные
- [ ] Черновик не дублируется

## 9. Сценарий без базы

Шаги:
- [ ] Начать новый ГНБ по объекту, где нет previous base

Проверка:
- [ ] Бот корректно работает без базы
- [ ] Черновик собирается с нуля
- [ ] Review честно показывает недостающие поля
- [ ] Подтверждение проходит только после заполнения обязательных данных

## 10. Поведение без активного черновика

Шаги:
- [ ] Без активного intake-черновика отправить обычный текст
- [ ] Отправить PDF
- [ ] Отправить Excel

Проверка:
- [ ] Бот ведёт себя как обычный fallback-ассистент
- [ ] Документы не попадают в intake-режим по ошибке
